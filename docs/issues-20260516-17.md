## 问题1: 排列三 API (sporttery.cn) 连续 567 限频

### 状态
⚠️ **未解决** — 自 2026-05-15 13:44 起排列三API持续返回 HTTP 567 限频，
期间偶有恢复（如 00:34 成功一次），但大部分时间仍失败。

### 现象
- 每次请求都返回 HTTP 567 "Server Error: Unknown Status"
- data_fetcher.py 的 3 次重试（间隔 5s/10s/15s）全部失败，耗时约 30 秒后放弃
- 受影响的运行时段（日志统计）:
  - 05-15 13:44~22:47: **约 30+ 次请求，全部 567**
  - 05-16 00:36~11:29: **约 15+ 次请求，全部 567**
  - 05-16 17:30~17:47: **约 10+ 次请求，全部 567** ← **17:30 cron 也失效**
  - 05-16 22:34~22:47: **全部 567**
  - 05-17 00:34~00:37: **全部 567**
- 唯一的例外: 05-17 00:34 的 `run_daily.py --strategy all` 排列三数据更新成功了（日志显示"✅ 排列三 数据更新"），
  但后续 00:37 的 `daily_review.py` 仍然 567 失败

### 影响
- **17:30 和 21:30 的 Hermes cron 预测推送中，排列三数据均使用前一天的缓存**
- `run_daily.py` 的数据更新步骤失败后会自动 warning 并继续使用旧数据，所以特征工程和评分仍能运行
- 但如果连续多天无法更新，排列三数据滞后天数会累积

### 可能的根因
- sporttery.cn API 前有腾讯 EdgeOne WAF 防护，连续请求（哪怕间隔 >30 秒）触发限频
- 该服务器 IP（阿里云广州节点）可能被列入临时黑名单
- 以往夜间（00:30+）容易恢复，但目前看 00:30~08:00 也连续失败

### 潜在修复方向
- 增加重试间隔（当前 5s/10s/15s 不够长，考虑 30s/60s/120s）
- 更换请求的 User-Agent / 添加随机延迟
- 寻找排列三的备用数据源（如东方财富/百度彩票/Lottery.gov.cn）
- 缩小 pageSize（从 30→5）减小触发概率
- 考虑在 `data_sources.yaml` 中为 pls 也添加备用源

## 问题2: 福彩3D zhcw 主源不稳定 + 曾返回错误号码

### 状态
⚠️ **未解决** — zhcw 的 `kaijiang.zhcw.com/zhcw/html/3d/list_N.html` 静态度量间歇返回 404，
同时此前已确认 zhcw 曾在 2026-05-16 返回 **错误号码**（954 而非真实的 846）。

### 现象
- **间歇 404**: 同一页面有时成功有时 404（CDN 缓存问题）
- **解析失败**: 有时 HTTP 200 但 `pd.read_html` 解析结果为 0 条记录
- **Vue 动态页面混淆**: 代码中曾有 URL 指向 `www.zhcw.com/kjxx/3d/`（Vue 动态页面），
  后来修复为 `kaijiang.zhcw.com/zhcw/html/3d/list_N.html`（静态表格），
  但旧页面访问会触发 "Vue动态渲染" warning
- **错误号码风险**: 即使 HTTP 200 + 解析成功，返回的号码可能不正确
  （原因是 zhcw 页面结构变化导致 `pd.read_html` 解析到错误行列）

### 影响
- 05-16 22:34 起 `daily_review.py` 的福彩3D数据已自动回退到东方财富源
- 但 `data_sources.yaml` 中 zhcw 仍标记为 primary 源，eastmoney 为 verify 且 `enabled: false`
- 如果 cron 调用时未激活东方财富 fallback 逻辑（取决于具体的脚本分支），仍可能使用 zhcw

### 已采取的缓解措施
- 05-16 08:34 起 data_fetcher 新增 `fetch_3d_from_eastmoney()` 备用源
- 05-16 08:34 起启用双源校验（检查两源最新期号号码一致性）
- 22:34 起主源失败自动回退到东方财富（日志: `⚠️ 主源(zhcw)失败，回退到东方财富`）

### 潜在修复方向
- **将东方财富设为福彩3D主源**，zhcw 降为备用/校验源
- or 继续保持双源自动校验，但将校验失败（号码不一致）时的行为从 warning 改为阻塞
- 当前 `data_sources.yaml` 配置需要修改：eastmoney 的 `enabled: false` → `enabled: true`，
  zhcw 的 `enabled: true` → `enabled: false`
