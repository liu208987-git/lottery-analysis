## 四、修复方案演变过程

### 第一轮（15:39）：软链接方案

> **结论：22:00 后被 Hermes 安全策略拦截，所有软链接无法执行**

**做法：** 把 `~/.hermes/scripts/` 下的旧副本替换为指向项目 `scripts/push/` 的软链接。

**失败原因：** Hermes no_agent 模式的 `script` 参数会在执行时检查文件的实际路径。软链接的 `readlink` 解析后指向 `~/.hermes/scripts/` 外部，触发安全拦截：
```
Blocked: script path resolves outside the scripts directory (/home/admin/.hermes/scripts): 'kl8_review_push.sh'
```

**影响范围：** 全部 5 个软链接都被拦截，包括：
- `lottery_review_push.sh`（21:35 和 22:05 两波复盘推送失败）
- `kl8_review_push.sh`（22:15 快乐8复盘推送失败）
- `lottery_predict_push.sh`（14:40 的预测推送在软链建立前已执行，未被拦截）
- `kl8_predict_push.sh`（14:50 的预测推送本来就失败）

### 第二轮（22:17）：实体文件复制方案

**做法：** 删除所有软链接，将项目 `scripts/push/` 下的 5 个 `.sh` 文件实际复制到 `~/.hermes/scripts/`。

**当前状态：**
```
~/.hermes/scripts/
├── kl8_check_push.sh        480 bytes  ✅ 实体文件（新增）
├── kl8_predict_push.sh      698 bytes  ✅ 实体文件（新增）
├── kl8_review_push.sh      1080 bytes  ✅ 实体文件（新增）
├── lottery_predict_push.sh  971 bytes  ✅ 实体文件（项目最新版）
└── lottery_review_push.sh  1460 bytes  ✅ 实体文件（项目最新版，带自动检测时间逻辑）
```

**后续维护方式：** 修改项目 `scripts/push/` 下的脚本后，必须手动执行：
```bash
cp scripts/push/*.sh ~/.hermes/scripts/
```
不再使用软链接。

### 关键教训

| 教训 | 说明 |
|:----|------|
| 🔴 软链接被 Hermes 安全策略拦截 | no_agent 模式的 `script` 参数解析软链接指向的目标路径，若在 `~/.hermes/scripts/` 外部则拒绝执行 |
| 🔴 下午的软链验证没覆盖 cron 实际执行 | 我在 15:39 手动跑 `~/.hermes/scripts/kl8_predict_push.sh` 成功了，但 Hermes cron 执行时有额外的安全扫描 |
| 🔴 `lottery_review_push.sh` 的旧版问题没有在软链阶段被发现 | 15:39 建软链时以为"软链=自动用新版"，没意识到 Hermes 会拦截，21:35 才暴露 |
