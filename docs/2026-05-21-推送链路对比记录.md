### 核心决策差异：为什么不选 GPT 的"绝对 command"方案

GPT 的方案是：**把 cron job 从 `script=xxx.sh` + `no_agent=true` 改成 `prompt=cd xxx && bash xxx.sh` 的 agent 模式**。

不选的原因：
1. **Hermes cronjob 工具没有 `command` 参数**——GPT 说的 `command = cd xxx && bash xxx.sh` 是 HERMES_CONFIG.md 的配置格式，不是 cronjob API 的参数
2. **改成 agent 模式会消耗 API token**——no_agent 模式不消耗 token，这是之前选用 no_agent 的核心原因
3. **软链接方案改动最小**——cron 配置一行不改，只需把 `~/.hermes/scripts/` 下的副本文件换成软链接

## 四、修复方案与修改内容

### 核心思路

**不改变 cron job 配置**（保留 `script=相对路径` + `no_agent=true`），而是把 `~/.hermes/scripts/` 下的**旧副本全部替换为软链接**，指向项目 `scripts/push/` 下的真实脚本。这样：
- cron 不需要改配置，仍然通过 `~/.hermes/scripts/xxx.sh` 执行
- 但实际执行的是项目文件，项目更新脚本后自动生效
- 不再有"副本过期"的问题

### 修改2：`lottery_review_push.sh` 增加自动检测时间逻辑（解决 23:10 传参问题）

**修改前（第12-15行）：**
```bash
FINAL_FLAG="--complete-only"
if [ "${1:-}" = "--final" ]; then
    FINAL_FLAG="--final-check"
fi
```

**修改后（第12-17行）：**
```bash
# 23:10 的 cron 无法传参（no_agent script 模式），改为自动检测时间
# 23 点后自动启用 --final-check
if [ "${1:-}" = "--final" ]; then
    FINAL_FLAG="--final-check"
elif [ "$(date +%H)" -ge 23 ]; then
    FINAL_FLAG="--final-check"
else
    FINAL_FLAG="--complete-only"
fi
```

**设计要点：**
- `--final` 手动传参仍然支持（优先级最高），GitHub Actions 或手动执行时可以传参
- 23 点后自动启用 `--final-check`，覆盖 23:10 的 cron 场景
- 否则默认 `--complete-only`（21:35/22:05 的默认行为）
